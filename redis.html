1. Redis はなぜ速いですか

Redis は主にメモリ上でデータを扱うため、ディスク I/O が中心の RDB より高速に読み書きできます。 さらにシンプルなデータ構造とイベントループ設計により、ロック競合が起きにくい点も効いています。

工程判断: 高頻度参照・低遅延要求の箇所に限定して使い、正本データは RDB に残す前提で設計します。

2. Redis と RDB はどう使い分けますか

RDB は業務データの正本、Redis は性能改善のための補助用途として分けるのが基本です。 例えば会員情報は RDB に保持し、参照頻度が高い項目のみ Redis にキャッシュします。

リスク管理: 責務を混ぜると整合性事故が起きやすいため、更新系は RDB 基準で統制します。

3. Redis に向いているデータは何ですか

高頻度参照で、多少の鮮度遅延を許容できるデータに向いています。 代表例はマスタデータ、セッション情報、レート制御カウンタ、一時状態管理です。

不向きな領域: 複雑検索や厳密トランザクションが必要な領域は RDB 側で処理します。

4. TTL はどう決めますか

TTL は「更新頻度」と「古いデータ許容度」の 2 軸で決めます。 整合性が重要なデータは短め、参照系で鮮度要求が低いものは長めに設定します。

実運用: TTL 単体ではなく、更新時の明示削除と組み合わせて鮮度ギャップを抑えます。

タグ一覧へ戻る

5. キャッシュ無効化はどうしますか

DB 更新後に該当キーを削除し、次回参照時に DB から再ロードする方式を採用します。 これにより更新直後の古いデータ残存時間を最小化できます。

加点ポイント: TTL だけに依存しない設計判断を示すと、実務理解が伝わりやすいです。

タグ一覧へ戻る

6. Redis は永続化できますか

可能です。RDB はスナップショット保存、AOF は更新操作ログの追記方式です。 要件に応じて復旧速度・データ損失許容を見ながら使い分けます。

工程判断: 永続化があっても、業務データ正本を Redis に寄せすぎない方が安全です。

7. Redis をセッション管理で使う理由は何ですか

セッションは高頻度参照のため Redis と相性が良く、複数サーバー間でも共有しやすいです。 TTL を設定すれば期限切れセッションを自動で整理できます。

実務観点: スケールアウト時のセッション一貫性と運用負荷の両面でメリットがあります。

8. Redis の注意点は何ですか

メモリ使用量の増加と、eviction 発生時の挙動管理が主な注意点です。 無制限に置くとコスト増と予期せぬデータ追い出しが起きるため、用途を限定します。

リスク管理: キー設計・容量上限・監視メトリクスを事前に決めておくことが重要です。

タグ一覧へ戻る

9. Redis でロックはできますか

可能です。例えば SETNX と有効期限を使って簡易ロックを実装できます。 ただし分散ロックではタイムアウト設計、解放漏れ、障害時の再取得競合に注意が必要です。

工程判断: 要件次第で DB 側の排他や別方式と比較し、過不足ない実装を選びます。

10. 何でも Redis に置けばよいですか

いいえ。Redis は強力ですが万能ではありません。 性能改善が必要な箇所へ限定し、DB・アプリ・Redis の責務を明確に分ける設計が安全です。

面接での締め方: 「性能要件と運用負荷を見て選ぶ」という判断軸を示すと高評価につながります。
